# RULES

## Non-negotiable
- NEVER RUN DESTRUCTIVE OPERATIONS WITHOUT EXPLICIT USER APPROVAL.
- NEVER FABRICATE RESULTS, TEST OUTCOMES, OR TOOL OUTPUTS.
- NEVER SELF-EDIT `RULES.md` AUTOMATICALLY.
- PREFER ROOT-CAUSE FIXES OVER SUPERFICIAL PATCHES.
- KEEP CHANGES MINIMAL, REVERSIBLE, AND WELL-TESTED.

## Safety
- VALIDATE ASSUMPTIONS BEFORE BROAD REFACTORS.
- FOR RISKY EDITS, SHOW THE PLAN AND ROLLBACK PATH FIRST.
- KEEP SECURITY AND DATA PRIVACY CONSTRAINTS EXPLICIT.

## Quality
- RUN RELEVANT CHECKS FOR TOUCHED CODE.
- IF CHECKS CANNOT RUN, SAY WHY AND PROVIDE MANUAL VERIFICATION STEPS.
- DO NOT ASSUME THAT ERRORS WERE PRESENT.
- DO NOT CLAIM A BUG, ERROR, OR REGRESSION WITHOUT EVIDENCE.
- DO NOT REPORT SUCCESS WITHOUT VERIFYING THE RESULT.
- DOCUMENT BEHAVIOR CHANGES IN USER-FACING SUMMARIES.
- WHEN ADDING TESTS, FIRST READ SIMILAR EXISTING TESTS AND MATCH THEIR STYLE.
- READ SIMILAR NEARBY CODE BEFORE INTRODUCING A NEW PATTERN.

## Version control
- BEFORE VERSION-CONTROL ACTIONS, FIRST DETERMINE WHETHER THE REPO USES GITBUTLER OR REGULAR GIT.
- IN A GITBUTLER REPO, KEEP THE WORKSPACE CLEAN AND UP TO DATE BEFORE CONFLICTS ARISE OR BEFORE VERSION-CONTROL ACTIONS.
- NEVER ASSUME UNASSIGNED CHANGES BELONG TO THE CURRENT TASK.
- NEVER PUSH THE GITBUTLER WORKSPACE BRANCH TO THE FORGE.

## Editing
- DO NOT MODIFY FILES UNLESS SPECIFIED BY THE USER.
- DO NOT CHANGE FILES OUTSIDE THE REQUESTED SCOPE WITHOUT EXPLICIT USER APPROVAL.
